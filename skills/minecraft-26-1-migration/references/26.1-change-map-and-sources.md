# 1.21.x → 26.1 Change Map and Sources

Research date: 2026-07-15. This reference separates upstream documentation, observed project implementation, and recommendations. Version numbers in announcements are historical snapshots; inspect the exact 26.1.x template and artifact metadata before pinning.

## Source priority

1. Exact target Fabric example branch or NeoForge MDK.
2. Official Fabric/NeoForge porting docs and release posts.
3. NeoForge vanilla primers, which Fabric also links for vanilla changes.
4. Actual target source and bytecode.
5. Mature project port diffs as architecture examples, not universal contracts.

Primers are high-level and non-exhaustive. They mostly cover vanilla internals, not every loader API.

## Documented target facts

### Java, Gradle, names, and mappings

- Minecraft/NeoForge/Fabric 26.1 development requires Java 25.
- Mojang stopped obfuscating Java Edition in 26.1. Official names and parameter names ship directly.
- `ResourceLocation` was renamed to `Identifier` at the 1.21.11 boundary.
- JSpecify replaces older nullability annotation usage in current official sources.
- Fabric 26.1 no longer officially supports Yarn: migrate to official names, remove the mappings dependency, switch to non-remapping Loom, use normal dependency configurations and `jar`, and use the `official` namespace in access wideners/class tweakers.
- NeoForge 26.1 requires modern Gradle/ModDevGradle; use the exact current MDK rather than announcement-era plugin pins.

### Fabric API official-name migration

Fabric API performed source-incompatible official-name renames not covered by mapping migration. Examples include world→level event names, `FabricDataOutput`→`FabricPackOutput`, tag/datagen provider renames, and directional networking names. Removed APIs/modules include `HudRenderCallback`, old conventional tags, loot API v2, and typical `TradeOfferHelper` use after data-driven villager trades.

Sources:

- Fabric API migration guide: https://docs.fabricmc.net/develop/porting/fabric-api
- IntelliJ migration map: https://docs.fabricmc.net/assets/develop/porting/fabric-api-26-1-migration-map.xml

### Registration and registries

Across 1.21.x, item/block properties gained resource-key requirements, block entity builder patterns changed, lookup names changed, and several 26.1 registries now store `MapCodec` directly rather than wrapper `*Type` objects. NeoForge target registration uses `DeferredRegister` and the mod event bus; registry values must not be queried while registration is in progress.

Sources:

- NeoForge registries: https://docs.neoforged.net/docs/concepts/registries
- 1.21.2 release/primer and 26.1 loot type unrolling primer section.

### Components, serialization, templates, and transfers

- Item behavior increasingly moved to data components in 1.21.5.
- Component values should be immutable, have stable equality, and define persistent and/or network codecs as needed.
- 1.21.6 migrated many direct `CompoundTag` methods to `ValueInput`/`ValueOutput`.
- 26.1 separates runtime mutable `ItemStack`/`FluidStack` from registry-independent immutable `ItemStackTemplate`/`FluidStackTemplate`.
- NeoForge 1.21.9 replaced old item/fluid/energy handlers with transactional resource/energy handler APIs.
- 26.1 introduces registry-context-aware `DataComponentInitializers` for holder defaults.

Sources:

- Fabric components: https://docs.fabricmc.net/develop/items/custom-data-components
- NeoForge attachments: https://docs.neoforged.net/docs/datastorage/attachments

### Networking

Both target ecosystems use typed `CustomPacketPayload`, a payload type, and `StreamCodec`; registry-aware codecs generally use `RegistryFriendlyByteBuf`. Direction, play/configuration phase, physical side, logical thread, and server validation must be explicit.

NeoForge documents default main-thread handling and custom payload limits of 1 MiB clientbound and less than 32 KiB serverbound.

Sources:

- NeoForge payloads: https://docs.neoforged.net/docs/networking/payload
- NeoForge stream codecs: https://docs.neoforged.net/docs/networking/streamcodecs
- NeoForge configuration tasks: https://docs.neoforged.net/docs/networking/configuration-tasks
- Fabric networking: https://docs.fabricmc.net/develop/networking

### Rendering and GUI

Rendering is the dominant cumulative rewrite:

- render-state extraction spread across entities, block entities, level rendering, and GUI;
- 1.21.4 added separate client item definition JSON;
- 1.21.5 introduced `RenderPipeline`/GPU abstractions and major model loading changes;
- 1.21.9 split level extraction and submission, including `SubmitNodeCollector`;
- 1.21.11 changed render-type and baked-quad/material details;
- 26.1 renames `GuiGraphics` to `GuiGraphicsExtractor`, screen render methods to extraction methods, and removes/replaces older item/block dispatcher paths;
- Fabric advises against raw OpenGL and is moving toward renderer abstractions compatible with future Vulkan work.

Sources:

- 26.1 primer rendering: https://github.com/neoforged/.github/blob/main/primers/26.1/index.md#even-more-rendering-changes
- NeoForge item models: https://docs.neoforged.net/docs/resources/client/models/items
- NeoForge model loaders: https://docs.neoforged.net/docs/resources/client/models/modelloaders
- Fabric rendering concepts: https://docs.fabricmc.net/develop/rendering/basic-concepts

### Datagen and resources

NeoForge split client/server gathering in 1.21.4 and changed provider APIs/resource formats over later releases. Fabric 26.1 renamed many provider/output types. Port providers first and regenerate data. Current Fabric setup uses `DataGeneratorEntrypoint`, the `fabric-datagen` metadata entrypoint, and Loom's Fabric API DSL. Include test structures and generated assets in migration scope.

Sources:

- NeoForge resources/datagen: https://docs.neoforged.net/docs/resources/
- Fabric datagen setup: https://docs.fabricmc.net/develop/data-generation/setup

### Physical side and source sets

Fabric's `splitEnvironmentSourceSets()` keeps common and physical-client code in one JAR while adding compile-time protection. Common gameplay belongs in `main`; rendering, screens, key mappings, and client handlers belong in `client`. Dedicated-server startup remains mandatory because source sets cannot detect every runtime initializer or Mixin problem.

Sources:

- https://docs.fabricmc.net/develop/loom/#split-sources
- https://docs.fabricmc.net/develop/getting-started/project-structure
- https://docs.fabricmc.net/develop/loader/fabric-mod-json#entrypoints

## Recommendations derived from the evidence

- Treat mapping conversion, build migration, mechanical names, and behavior changes as distinct passes.
- Keep old/new workspaces side by side and compare corresponding vanilla usage.
- Prefer typed, coarse adapters over scattered loader checks or permanent regex rewrites.
- Store item definitions as templates and instantiate mutable stacks at runtime boundaries.
- Make rendering and optional integrations independent workstreams.
- Keep port work on an explicit branch and define how fixes flow from the maintained old line.
- Require a stabilization tail after the nominal port commit.
- Test data assets, dedicated servers, actual Mixins, old worlds, and malformed client packets; compilation is only the first gate.
